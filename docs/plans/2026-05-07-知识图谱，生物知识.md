# 知识图谱，生物知识接入方案

## 目标

为 MadAI 的 `protocol`、trial retrieval 和 eligibility assessment 增加可审计的医学背景知识通道。

本方案不试图自建一个完整医学百科，而是把现成的医学术语库、健康百科、药物标签、文献证据和生物医学知识图谱接成一组工具。它们用于辅助：

1. trial query profile 扩展，例如疾病别名、MeSH 词、药物标准名和适应症。
2. inclusion/exclusion criteria 解析，例如把自然语言标准归一化到疾病、药物、检查、基因、表型和治疗概念。
3. eligibility assessment 背景解释，例如复杂治疗线数、CNS metastases、organ function threshold、contraindication 等。
4. treatment / trial recommendation bundle 的证据来源补充。
5. reporter 输出时提供可引用的医学背景和来源链接。

边界约束：

1. 医学知识通道不能直接替代 trial 原文和患者证据做 eligibility 结论。
2. `medical_knowledge_bundle` 只提供背景、归一化、候选概念和证据链接。
3. 最终 criterion label 仍必须来自 trial criterion 与患者证据的比对。
4. 所有外部知识都应保留来源、版本、查询词和时间戳，避免不可追溯的“模型常识”进入判断链路。

## 核心结论

第一版建议优先接入 NLM / NIH / FDA 体系的稳定知识源：

```text
MedlinePlus       医学百科和健康主题解释
MeSH              疾病/主题词归一化和 PubMed 查询扩展
RxNorm            药物名、成分、RxCUI 归一化
DailyMed          药品标签、适应症、禁忌、警告、不良反应
PubMed/E-utilities 文献检索
PubTator          文献和文本中的疾病、药物、基因等实体标注
```

第二阶段再接：

```text
UMLS              多本体术语中枢，覆盖 MeSH、RxNorm、SNOMED CT、ICD、LOINC 等
PrimeKG           生物医学知识图谱，覆盖疾病、药物、基因、表型、通路等关系
BioPortal         多 ontology API fallback
ToolUniverse      agent 工具层和 biomedical API 编排参考
```

不建议第一版直接做：

1. 全量 UMLS 本地镜像和完整 license 管理。
2. SNOMED CT 全量推理。
3. 大规模图数据库生产化部署。
4. 依赖 Wikipedia/Wikidata 作为主医学知识源。
5. 用一个 LLM prompt 代替术语归一化、药品标签查询和文献证据检索。

## 推荐架构

新增一个轻量的 `MedicalKnowledgeService`，它不是一个 agent，而是一组可被 `protocol` subgraph 调用的工具。

建议模块边界：

```text
agent/medical_knowledge/
  __init__.py
  types.py
  service.py
  entity_linking.py
  medlineplus.py
  mesh.py
  rxnorm.py
  dailymed.py
  pubmed.py
  pubtator.py
  primekg.py
  cache.py
```

第一版工具接口：

```python
medical_entity_link(text)
  -> normalized concepts, aliases, source vocabularies, confidence

medlineplus_lookup(query_or_code)
  -> health topic summary, aliases, MeSH terms, related topics, source URL

mesh_lookup(query)
  -> MeSH descriptors, tree numbers, entry terms, PubMed query expansion terms

rxnorm_lookup(drug_name)
  -> RxCUI, normalized drug names, ingredients, dose forms, related names

dailymed_label_lookup(drug_name_or_rxcui)
  -> indications, contraindications, warnings, adverse reactions, label URL

pubmed_evidence_search(query, concepts)
  -> PMID, title, abstract snippets, publication date, source URL

pubtator_annotate(text_or_pmids)
  -> disease, chemical, gene, species, mutation, cell line annotations
```

第二版工具接口：

```python
primekg_neighbors(concept, node_types=None, max_results=50)
  -> related diseases, drugs, phenotypes, genes, pathways

primekg_paths(source_concept, target_concept, max_hops=2)
  -> short evidence paths and relation types

primekg_disease_expansion(disease)
  -> aliases, phenotypes, related diseases, genes, drugs

primekg_drug_disease_edges(drug, disease)
  -> relation candidates for indication, contraindication, trial query expansion
```

建议输出统一成 `MedicalKnowledgeBundle`：

```json
{
  "schema_version": 1,
  "queries": [
    {
      "query_id": "",
      "purpose": "trial_query_expansion | criterion_interpretation | drug_safety | evidence_search",
      "raw_query": "",
      "normalized_query": "",
      "sources_requested": ["mesh", "medlineplus", "rxnorm", "dailymed", "pubmed"],
      "created_at": ""
    }
  ],
  "concepts": [
    {
      "concept_id": "",
      "source": "mesh | rxnorm | umls | medlineplus | primekg",
      "code": "",
      "preferred_name": "",
      "aliases": [],
      "semantic_type": "",
      "score": 0.0,
      "source_url": ""
    }
  ],
  "knowledge_items": [
    {
      "item_id": "",
      "source": "",
      "title": "",
      "summary": "",
      "evidence_type": "encyclopedia | terminology | drug_label | literature | graph_edge",
      "concept_ids": [],
      "source_url": "",
      "retrieved_at": ""
    }
  ],
  "knowledge_gaps": [
    {
      "gap_type": "",
      "description": "",
      "impact": "low | medium | high"
    }
  ]
}
```

## 与 Protocol 的关系

`protocol` 内部应把医学知识作为单独模块接入：

```text
run_protocol_subgraph
  -> case_protocol_profile_agent
  -> trial_retrieval_module
  -> medical_knowledge_retrieval_module
  -> calculator_evidence_integration_module
  -> eligibility_assessment_module
  -> protocol_selection_module
  -> recommendation_bundle_module
```

`medical_knowledge_retrieval_module` 的输入：

1. `structured_case_json`
2. `calculation_results`
3. `trial_retrieval_bundle.candidate_ranking`
4. top trials 的 inclusion/exclusion criteria
5. trial interventions、conditions、keywords、biomarkers

`medical_knowledge_retrieval_module` 的输出：

```json
{
  "medical_knowledge_bundle": {
    "schema_version": 1,
    "case_concepts": [],
    "trial_concepts": [],
    "criterion_concepts": [],
    "drug_knowledge": [],
    "disease_knowledge": [],
    "literature_evidence": [],
    "query_expansions": [],
    "knowledge_gaps": []
  }
}
```

关键原则：

1. 对粗召回，医学知识主要产生 query expansion。
2. 对精召回，医学知识主要产生 normalized concepts 和 aliases。
3. 对 eligibility assessment，医学知识主要解释 criterion 中的概念，不直接下结论。
4. 对 treatment recommendation，医学知识主要提供药物标签、适应症、禁忌、文献和图谱背景。
5. 对 reporter，医学知识必须以 source URL 和 evidence item 的形式出现，避免无来源医学解释。

## 第一阶段：轻量百科与术语归一化

目标是在不引入复杂 license 和图数据库的情况下，先提升 trial 检索和 eligibility 解析。

### MedlinePlus

用途：

1. 疾病、症状、检查和治疗的健康主题解释。
2. 获取 aliases、MeSH terms、related topics。
3. 给 reporter 提供面向临床用户可读的背景链接。

优先使用：

1. MedlinePlus Health Topic XML 本地索引。
2. MedlinePlus Connect 在线查询。

注意：

1. MedlinePlus 的疾病/健康主题适合作为百科背景。
2. 药品信息不建议依赖 MedlinePlus Connect 的 ASHP 内容做本地 ingest，因为存在版权限制。
3. 药物标签和禁忌优先用 DailyMed。

### MeSH

用途：

1. 疾病和医学主题标准化。
2. PubMed 查询扩展。
3. Trial condition 别名扩展。
4. 与 PubMed 文献证据对齐。

实现方式：

1. 第一版可使用 MeSH RDF / lookup API。
2. 本地可缓存 descriptor、entry terms、tree numbers。
3. 对 trial query profile 增加 MeSH descriptor 和 entry terms。

### scispaCy

用途：

1. biomedical NER。
2. abbreviation detection。
3. entity linking 到 MeSH、RxNorm、UMLS 等候选概念。

建议第一版先用 scispaCy 做候选生成器，不把它当成最终医学判断器。

输出要求：

```json
{
  "text": "",
  "span": [0, 0],
  "preferred_name": "",
  "source": "mesh | rxnorm | umls",
  "code": "",
  "score": 0.0,
  "aliases": []
}
```

## 第二阶段：药物知识

药物知识应独立于疾病百科实现，因为 trial eligibility 里药物相关标准非常常见：

1. prior therapy
2. concomitant medication
3. investigational agent exposure
4. contraindicated medication
5. drug class
6. washout period
7. adverse event / toxicity history

### RxNorm

用途：

1. 药物名标准化。
2. RxCUI 查询。
3. ingredient / brand / clinical drug 关系。
4. 药物同义词和 spelling variant 归一化。

建议：

1. 在线调用 RxNav RxNorm API。
2. 高频药物名做本地缓存。
3. trial interventions 和 patient medications 都映射到 RxCUI。

### DailyMed

用途：

1. 药品标签。
2. indications and usage。
3. contraindications。
4. warnings and precautions。
5. adverse reactions。
6. boxed warning。

建议：

1. 通过 drug name 或 RxCUI 查询 SPL labels。
2. 只抽取与当前任务相关的 label sections。
3. 保存 label URL、SPL set ID、retrieved_at。

注意：

DailyMed label 很长，不应全量塞进 prompt。应先结构化抽 section，再做短摘要或片段检索。

## 第三阶段：文献证据

文献证据不应作为第一轮 eligibility 的硬依赖，但应作为复杂判断和推荐解释的支持。

### PubMed E-utilities

用途：

1. 根据 MeSH、疾病、药物、基因和 trial intervention 查询文献。
2. 提供 evidence search 的可追溯来源。
3. 给 reporter 输出 PMID 和来源链接。

建议：

1. `esearch` 获取 PMID。
2. `efetch` 获取摘要。
3. `elink` 查询相关文献。
4. 查询式应由 MeSH terms、drug names、condition 和 trial context 构造。

### PubTator

用途：

1. 对 PubMed 摘要或本地文本做实体标注。
2. 提取 disease、chemical、gene、mutation 等概念。
3. 对 evidence snippet 做结构化 concept linking。

建议：

PubTator 作为文献证据的 annotation 层，不作为唯一检索层。

## 第四阶段：生物医学知识图谱

知识图谱适合用在 query expansion、疾病-表型扩展、药物-疾病关系和复杂概念解释上。

### PrimeKG

优先级最高。

用途：

1. disease、drug、gene、phenotype、pathway、exposure 的关系查询。
2. rare disease 和复杂疾病相关概念扩展。
3. drug-disease 关系候选。
4. biomarker / gene 相关 trial query expansion。

建议实现：

1. 第一版把 PrimeKG CSV 下载到本地。
2. 用 DuckDB 或 SQLite 建本地轻量索引。
3. 不急于上 Neo4j。
4. 提供 neighbors 和 short paths 两类查询接口。

示例查询：

```text
患者疾病: metastatic non-small cell lung cancer
扩展:
  MeSH descriptor
  disease aliases
  related phenotypes
  associated genes
  related drugs
  trial intervention aliases
```

### Hetionet

用途：

1. drug repurposing / disease network 原型参考。
2. 图谱格式和关系建模参考。

优先级低于 PrimeKG。

### BioPortal

用途：

1. 多 ontology 查询 fallback。
2. ontology recommender。
3. annotator。

注意：

BioPortal 需要 API key，且依赖外部服务，不建议作为高频主链路。

## Agent 设计可借鉴项目

### TxAgent / ToolUniverse

最值得借鉴的是工具层，不是具体 prompt。

可借鉴点：

1. biomedical tools 统一注册。
2. tool discovery。
3. agent 根据任务选择 API / 数据库 / scientific package。
4. 工具输出保持结构化，最后再由模型综合。
5. 对复杂 biomedical reasoning 使用外部工具而不是模型常识。

对 MadAI 的启发：

```text
MedicalKnowledgeService
  -> tool registry
  -> source-specific adapter
  -> cache
  -> structured result
  -> protocol agent synthesis
```

### MedAgents / MDAgents

这类项目更适合借鉴多 agent 协作模式：

1. 根据复杂度决定 solo / group collaboration。
2. 对复杂医学问题使用 moderator review。
3. 把外部医学知识作为复杂判断的输入。

对 MadAI 的启发：

1. 简单 eligibility 标准走规则。
2. 复杂治疗线数、基因组、模糊时间窗走 LLM reviewer。
3. reviewer 必须消费 `medical_knowledge_bundle` 和 patient evidence spans。
4. moderator 只审查证据链和标签一致性，不自由生成无来源医学结论。

## 数据缓存与版本管理

所有外部知识必须缓存并版本化。

建议缓存字段：

```json
{
  "cache_key": "",
  "source": "",
  "source_version": "",
  "query": "",
  "normalized_query": "",
  "response_hash": "",
  "retrieved_at": "",
  "expires_at": "",
  "source_url": ""
}
```

建议策略：

1. MedlinePlus XML、MeSH、PrimeKG 使用本地版本化快照。
2. RxNorm、DailyMed、PubMed、PubTator 使用在线查询加本地缓存。
3. 缓存结果进入 `outputs/` 或独立 data 目录，不进入 git。
4. 每个 final bundle 保留 source 和 retrieved_at。

## 与 Trial 检索参数的关系

医学知识接入后，trial 检索可以分成三类 query expansion：

```text
case_terms
  患者原始诊断、症状、药物、基因、检查

normalized_terms
  MeSH、RxNorm、UMLS、MedlinePlus aliases

graph_terms
  PrimeKG related disease、phenotype、gene、drug
```

建议用于粗召回：

```text
coarse query = condition + diagnosis aliases + MeSH entry terms + intervention aliases
```

建议用于精召回：

```text
fine query = trial criterion terms + normalized patient concepts + key evidence terms
```

建议用于 eligibility assessment：

```text
criterion interpretation context =
  original criterion
  normalized concepts
  aliases
  drug label snippets
  disease/topic summary snippets
  patient evidence spans
```

注意：

医学知识扩展会提高召回，但也会引入噪声。建议先只把高置信别名和标准术语用于 coarse retrieval，不把 PrimeKG 远邻关系直接加入主查询。

## 实施顺序

第一批提交：

1. 新增 `agent/medical_knowledge/types.py`。
2. 新增 `MedicalKnowledgeBundle`、`MedicalConcept`、`KnowledgeItem`、`KnowledgeQuery` 数据结构。
3. 新增 `mesh.py`、`rxnorm.py`、`dailymed.py`、`pubmed.py` 的轻量 adapter skeleton。
4. 新增 `service.py` 统一调度。
5. 在 `protocol_node` 中保留 `medical_knowledge_bundle` 输出字段，但第一批允许为空或只包含 query expansion。

第二批提交：

1. 接入 MeSH / MedlinePlus 本地或在线查询。
2. 接入 RxNorm 和 DailyMed 在线查询。
3. 为 trial retrieval query profile 增加 normalized terms 和 aliases。
4. 增加缓存。
5. 增加单元测试覆盖 query normalization 和 bundle schema。

第三批提交：

1. 接入 PubMed E-utilities。
2. 接入 PubTator annotation。
3. 在 complex criterion reviewer 中消费文献证据。
4. reporter 输出 PMID / source URL。

第四批提交：

1. 下载和索引 PrimeKG。
2. 实现 `primekg_neighbors` 和 `primekg_paths`。
3. 只在研究评估模式中启用 graph expansion。
4. 评估 graph expansion 对 coarse recall 和 precision 的影响。

## 评估指标

检索侧：

1. coarse recall 是否提升。
2. fine rerank top-k 中 trial condition / intervention 命中率是否提升。
3. query expansion 后候选噪声是否增加。

eligibility 侧：

1. criterion entity linking 命中率。
2. drug criterion 的 RxNorm / DailyMed 匹配率。
3. complex criterion 的 `unknown` 是否减少。
4. `met` / `not_met` 判断是否仍有证据支持。

系统侧：

1. 每次 protocol run 的外部 API 调用数。
2. 缓存命中率。
3. p95 latency。
4. bundle 体积。
5. source URL 完整率。

## 风险与限制

### License

UMLS 免费但有 license 约束。SNOMED CT、部分药品信息和第三方 ontology 也可能有分发限制。

第一版建议避开 license 最复杂的路径：

```text
先用 MeSH + MedlinePlus + RxNorm API + DailyMed + PubMed
后续再接 UMLS / SNOMED CT
```

### 医学准确性

百科、图谱和文献都可能过时或不适用于个体患者。

因此：

1. `medical_knowledge_bundle` 不能直接输出临床建议。
2. trial eligibility 必须以 trial 原文和患者证据为主。
3. 药物禁忌优先引用 DailyMed label。
4. 文献证据必须带 PMID / URL。

### 噪声控制

图谱邻居和 ontology aliases 很容易扩展过度。

建议：

1. query expansion 默认只使用同义词、标准词和高置信 entry terms。
2. graph terms 默认只作为 rerank feature 或 reviewer context。
3. 对每类 expansion 记录来源和权重。

### 延迟

在线 API 过多会拖慢 protocol。

建议：

1. 高频源本地缓存。
2. trial coarse retrieval 阶段只用本地索引。
3. DailyMed / PubMed 在 top candidates 阶段再查。
4. 研究模式可以放宽调用数量，在线模式应限制调用预算。
